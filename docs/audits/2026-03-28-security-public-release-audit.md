# Gameplan Turbo Security And Public Release Audit

Date: 2026-03-28

Method:
- Static review of bridge scripts, OAuth flow, provider persistence, export/download code, and release-facing docs/config.
- Pattern scans over the current tracked tree for secrets, home-directory paths, localhost references, and tool-login surfaces.
- Git-history scan for home-directory paths and common secret patterns.
- Baseline checks:
  - `corepack pnpm typecheck` passed
  - `corepack pnpm test -- --runInBand` passed (`34` files, `145` tests)
  - `corepack pnpm build` passed
- Limited manual browser verification:
  - Settings renders the provider/connect flows as expected.
  - Offline Claude bridge produces runtime console errors in Settings when status checks fire.

## Release Recommendation

Status: **Not ready to open-source publicly yet**

Release blockers:
1. Local bridges trust any localhost origin and have no caller authentication.
2. Current tracked docs still contain an absolute machine path, and older commits contain the same path in scripts and UI copy.
3. `.qwen/settings.json` is still tracked even though `.gitignore` excludes it.
4. CSP `connect-src` is out of sync with the provider list, so some advertised providers will not work in production.

Recommended remediation order:
1. Bridge hardening
2. Public-repo scrub of current tree and git history
3. CSP / provider allowlist fix
4. OAuth hardening and storage hardening follow-up

## What Already Looks Good

- No live API keys, bearer tokens, or private-key blocks were found in the current tracked tree from the pattern scan used in this audit.
- The bridge scripts pass prompts to the CLI via stdin or argument arrays with `spawn`, so there is no obvious shell-interpolation injection path from prompt content.
- App JSON export masks provider keys instead of exporting them raw.
- Planning package export does not include provider configs or stored API keys.
- Credential JSON export omits credential values.

## Confirmed Security Findings

### 1. Local bridges accept any localhost origin and do not require a caller secret
- Severity: high
- Release blocker: yes
- Impact:
  - Any page running on `http://localhost:*` or `http://127.0.0.1:*` can call the bridge.
  - If the user is logged into Codex or Claude Code locally, a malicious local page or app can trigger arbitrary prompt execution, read responses, or open login windows.
- Exploitability / repro conditions:
  - Requires the user to visit or run an attacker-controlled local page/app on localhost.
  - No per-session secret, nonce, or app-origin pinning is required to reach `/generate` or `/auth/open-login`.
- Evidence:
  - `scripts/codex-bridge.mjs:17-31`
  - `scripts/codex-bridge.mjs:284-352`
  - `scripts/claude-bridge.mjs:17-31`
  - `scripts/claude-bridge.mjs:240-302`
- Notes:
  - The current regex allows any localhost port.
  - Requests with no `Origin` header are also allowed through the origin gate.
- Recommended fix:
  - Pin bridge access to the exact app origin(s) instead of any localhost origin.
  - Require a per-session shared secret or nonce header generated by the launcher/app.
  - Reject missing `Origin` headers for browser-facing endpoints unless there is an explicit trusted native caller path.

### 2. CSP allowlist is incomplete for the supported provider set
- Severity: medium
- Release blocker: yes
- Impact:
  - `OpenRouter`, `GLM / Zhipu`, `Kimi / Moonshot`, and `MiniMax` are in the provider catalog, but their API domains are not present in the current CSP `connect-src`.
  - Production builds can advertise providers that the browser policy blocks.
- Exploitability / repro conditions:
  - Triggered whenever a user tries to use one of the newly added hosted providers in production.
- Evidence:
  - `index.html:8`
  - `src/services/ai/providers/openrouterProvider.ts:7`
  - `src/services/ai/providers/glmProvider.ts:7`
  - `src/services/ai/providers/moonshotProvider.ts:7`
  - `src/services/ai/providers/minimaxProvider.ts:7`
- Recommended fix:
  - Synchronize CSP `connect-src` with the actual provider catalog and OAuth exchange endpoints.
  - Keep the allowlist narrow and explicit; do not work around this by broadly loosening `connect-src`.

### 3. OpenRouter OAuth flow uses PKCE but does not use a state parameter
- Severity: medium
- Release blocker: no
- Impact:
  - The OAuth callback is weaker than standard best practice against callback confusion/CSRF-style mix-ups.
  - The PKCE verifier also remains in `sessionStorage` until a successful token exchange removes it.
- Exploitability / repro conditions:
  - Limited by PKCE and same-origin `postMessage`, so this is not a trivial takeover.
  - Still a meaningful hardening gap for a public release.
- Evidence:
  - `src/lib/ai/openRouterOAuth.ts:35-57`
  - `src/lib/ai/openRouterOAuth.ts:83-91`
  - `src/lib/ai/openRouterOAuth.ts:122-157`
  - `src/lib/ai/openRouterOAuth.ts:161-178`
- Recommended fix:
  - Add a cryptographic `state` value alongside PKCE and verify it in the callback.
  - Clear the PKCE verifier on popup-close/error paths, not only on successful exchange.

### 4. Provider API keys are stored in plaintext IndexedDB
- Severity: medium
- Release blocker: no, but needs an explicit product/security decision
- Impact:
  - Saved provider keys are stored raw in the browser database.
  - Any future XSS, compromised app origin, malicious extension, or local browser-profile compromise can expose all saved provider keys.
- Exploitability / repro conditions:
  - Requires local compromise of the browser profile or code execution in the app origin.
- Evidence:
  - `src/hooks/useAIProviders.ts:65-93`
  - `src/lib/db-schema.ts:23-24`
  - `src/lib/db-schema.ts:39`
- Recommended fix:
  - Prefer OS-keystore/native-secret storage for desktop-distributed builds.
  - If that is not available yet, add clear user-facing disclosure that keys are stored locally and unencrypted, plus a session-only mode.

### 5. Dormant Ship feature stores arbitrary credentials in plaintext browser storage
- Severity: medium
- Release blocker: no, because it is not routed today
- Impact:
  - The hidden `Ship` surface can act like a generic secret store: arbitrary credential values are stored, revealed, and copied from browser storage.
  - If this page is routed back into the product later without hardening, it becomes a direct secret-management risk.
- Exploitability / repro conditions:
  - Not reachable through the current workspace navigation.
  - Risk becomes active if the page is reintroduced or linked later.
- Evidence:
  - `src/pages/workspace/ShipPage.tsx:42-47`
  - `src/pages/workspace/ShipPage.tsx:134-156`
  - `src/hooks/useShip.ts:111-135`
  - `src/lib/db-schema.ts:27-28`
  - `src/lib/db-schema.ts:42-43`
- Recommended fix:
  - Remove this feature from the open-source branch until it has a stronger storage model and threat model.
  - At minimum, keep it unreachable and clearly mark it as experimental/internal.

## Public Repo / PII Findings

### 6. Absolute local machine path is still present in a tracked audit doc
- Severity: high
- Release blocker: yes
- Impact:
  - The public tree still exposes your local workspace path directly in a tracked file.
- Evidence:
  - `docs/audits/2026-03-27-gameplan-turbo-end-to-end-audit.md:5`
- Recommended fix:
  - Remove or sanitize the path in the current tree before publishing.

### 7. The same absolute local path is present in git history, including scripts and UI copy
- Severity: high
- Release blocker: yes
- Impact:
  - Even if the current tree is cleaned up, a public repo still exposes earlier commits with hardcoded home-directory paths in scripts and onboarding/settings copy.
- Evidence:
  - `git grep '/Users/<username>/' $(git rev-list --all)` found:
    - `docs/audits/2026-03-27-gameplan-turbo-end-to-end-audit.md`
    - older `scripts/launch-preflight-game-os.sh`
    - older onboarding/provider-card UI instructions
- Recommended fix:
  - If this repository is going public as-is, rewrite or squash history before publishing.
  - If preserving history matters, publish from a cleaned public branch/repo instead of this full private history.

### 8. `.qwen/settings.json` is tracked even though `.gitignore` now excludes `.qwen/`
- Severity: medium
- Release blocker: yes
- Impact:
  - Local tool-specific permission settings are shipping in the tracked tree.
  - This makes the repo feel tailored to one environment and can expose workflow-specific allowances.
- Evidence:
  - `.qwen/settings.json:1-9`
  - `.gitignore` excludes `.qwen/`, but `git ls-files '.qwen/*'` still returns the tracked file.
- Recommended fix:
  - Remove `.qwen/settings.json` from the index and ensure the ignore rule is effective for future commits.

### 9. Current public docs still read as partly personal/internal rather than open-source neutral
- Severity: low
- Release blocker: no
- Impact:
  - The repo still contains personal/internal narrative docs and private-audit style material that make the project feel like a personalized working folder rather than a clean public OSS release.
- Evidence:
  - `docs/behind-the-scenes/*`
  - `docs/audits/*`
  - installation and internal prompt docs still assume a local author workflow in places
- Recommended fix:
  - Decide which docs are meant for public readers.
  - Move internal notes/audits behind a private docs boundary or remove them from the public branch.

## Export / Data-Handling Notes

- **App JSON export**: provider keys are masked, not exported raw.
  - Evidence: `src/lib/appData.ts:18-35`, `src/lib/appData.ts:109-135`
- **Planning package export**: does not include provider configs or provider secrets.
  - Evidence: `src/lib/planningPackageExport.ts:202-255`
- **Credential export JSON**: omits stored credential values.
  - Evidence: `src/hooks/useShip.ts:176-211`
- **Caveat**:
  - Planning package export does include raw vault files by design.
  - If a user uploads sensitive files into Vault, exporting the planning package will include those files too.

## Follow-Up Remediation Priority

1. Add bridge caller authentication and tighten origin handling.
2. Scrub current tree and git history of local paths and tracked local-config files.
3. Fix CSP allowlist drift for all supported providers.
4. Add OAuth `state` and better PKCE cleanup.
5. Decide whether plaintext local key storage is acceptable for the public release or whether the app needs a keystore/session-only mode.
